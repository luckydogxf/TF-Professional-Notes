# 不足之处有哪些？

1. 状态文件本地化导致单点故障风险‌
Terraform 的状态文件（terraform.tfstate）是维护云资源状态的核心文件。若该文件仅存储于本地，一旦发生丢失或损坏，Terraform 将无法准确识别当前云环境的真实状态，从而导致无法有效追踪和管理资源，甚至引发误操作。

2. 缺乏团队协作机制‌
在多人协作场景中，若多个开发者同时操作同一份状态文件，容易引发冲突。例如，一人创建 EC2 实例，另一人创建 S3 存储桶，由于状态文件无法共享与并发编辑，将难以实现高效协同。

3. 代码复用性差，难以支撑多业务单元管理‌
若需为不同业务单元维护多套 Terraform 配置，通常只能通过复制粘贴代码并修改变量值的方式实现。这种方式不仅增加了维护成本，也容易引入人为错误。

4. 环境隔离与配置管理困难‌
对于 Dev、Staging 和 Production 等多个环境，若采用相同配置文件并手动切换变量，会导致配置混乱、难以统一管理。

5. 模块化与结构化不足‌
所有资源定义集中在一个 main.tf 文件中，缺乏模块化设计，难以实现资源的解耦与复用。例如，在创建 VPC 及其子网时，若未将 VPC 与子网资源拆分为独立模块，将导致配置文件臃肿，难以维护和扩展。
